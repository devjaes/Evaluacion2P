JAIR MERA


PROCESO ACS
  • Identificación de Objetos en la Configuración del Software:

    Selección y control de los elementos de software que se deben gestionar a lo largo del ciclo de vida del proyecto.

  • Control de Cambio:

    Procedimientos y mecanismos para la gestión y aprobación de cambios en el software, asegurando que se realicen de manera controlada y documentada.

  • Control de Versión:

    Gestión de las diferentes versiones del software y cómo se manejan las iteraciones del mismo.

  • Auditoría de Configuración y Reporte de Estado:

    Las auditorías verifican la conformidad con los requisitos y estándares, mientras que los reportes de estado proporcionan una visión actualizada del progreso del proyecto.

  • Auditoría de la Configuración:

    Importancia de las revisiones técnicas y auditorías en la configuración del software para asegurar una implementación adecuada de los cambios y mantener la integridad del producto.

  • Reporte de Estado:

    El reporte de estado de la configuración responde preguntas claves sobre los cambios realizados, quién los hizo, cuándo y qué más podría afectarse.

  • Gestión de Lanzamientos:

    Proceso para garantizar la entrega adecuada de productos (software, documentación, materiales auxiliares) a la parte receptora designada.


FLUJO DE TRABAJO DE GITFLOW
  • main:

    La rama main es la rama principal del proyecto, en ella se encuentran los archivos que se encuentran en producción.

  • develop

    La rama develop es la rama de desarrollo del proyecto, en ella se encuentran los archivos que se encuentran en desarrollo incluyendo las features que se van realizando.

  • feature/{nombre-feature}

    La rama feature es la rama de desarrollo de una nueva funcionalidad del proyecto, en ella se encuentran los archivos que se encuentran en desarrollo.

  • release/{nombre-release}

    La rama release es la rama de desarrollo de una nueva versión del proyecto, una vez finalizada la versión se integra a main.

  • hotfix/{nombre-hotfix}

    La rama hotfix es la rama donde se corrigen errores de la rama main, despues de corregir los errores se integra a main y develop.

  • Flujo:

    El flujo de trabajo de Gitflow es el siguiente:

    1. Se crea la rama develop a partir de main.
    2. Se crea la rama feature/{nombre-feature} a partir de develop. Una vez finalizada la funcionalidad se integra a develop.
    3. Se crea la rama release/{nombre-release} a partir de develop.
    4. Se crea la rama hotfix/{nombre-hotfix} a partir de main.

FLUJO DE PROCESO GESTIÓN DE CAMBIO
  • Envío:

    Inicio del proceso con la presentación de la solicitud de cambio.

  • Planificación:

    Análisis del impacto, elaboración del plan de implementación y plan de reversión (rollback) si es necesario.

  • Aprobación:

    Evaluación de la solicitud por el Comité Asesor de Cambios (CAB) para su aceptación o rechazo.

  • Implementación:

    Ejecución de las tareas necesarias para llevar a cabo el cambio aprobado.

  • Revisión:

    Inspección post-implementación para asegurar que los cambios cumplen con los requisitos.

  • Cierre:

    Conclusión y documentación formal del cambio implementado.
